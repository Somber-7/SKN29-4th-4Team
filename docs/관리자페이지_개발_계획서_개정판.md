# 관리자 페이지 개발 계획서 (개정판 · 운영 기능 확장)

작성일: 2026-07-06
전신 문서: `docs/관리자페이지_개발_계획서_최종.md` (BrowserRouter 전환·별도 번들·서버 RBAC 등 **확정 사항은 그대로 승계**)
기준 코드: `frontend/src/app/*`, `webapp/naming/*`(빈 스텁·그린필드), `deploy/nginx/*`
스택: React 18 + TS + Vite + Tailwind v4 / **Django 6.0 + Django Ninja**(관리자 API) / FastAPI / nginx 단일 도메인 / Docker Compose / AWS EC2 · RDS PostgreSQL · S3 / Neo4j · ChromaDB

> 백엔드 프레임워크 결정: 관리자 REST API는 **Django Ninja**(Pydantic 기반)로 확정한다. 백엔드가 그린필드라 전환 비용이 없고, 이미 FastAPI로 Pydantic을 쓰고 있어 학습곡선이 낮으며, 검증·직렬화 자동화 + OpenAPI 문서로 40~50개 관리자 엔드포인트의 보일러플레이트를 제거한다. (순수 FBV 대비 근거: §15.5)

> 본 개정판은 전신 문서의 **아키텍처·보안 결정을 뒤집지 않는다.** 라우팅(BrowserRouter), 관리자 별도 번들(`admin.html`/`main.admin.tsx`, `/manage/`), `/api/admin/*` + `@admin_required`, 관리자 세션 분리(`mgAdmin`), 초기 계정 커맨드·강제 비번 변경·로그인 잠금은 모두 **유지 전제**다. 이 문서는 그 위에 **운영 기능(사용자 관리·게시물·공지/문의/FAQ·통계·가입 승인·감사 로그 등)** 을 얹는다.

## 2. 유지되는 확정 사항 (재확인 — 이 문서에서 변경하지 않음)

- 라우팅: **BrowserRouter**, nginx SPA 폴백(`try_files`), 구 `/#/` 링크 하위호환.
- 관리자 프론트: **별도 번들**(`admin.html` + `main.admin.tsx`, `basename="/manage"`), 사용자 번들에 관리자 코드 **0바이트**, `/manage/` nginx location + IP 화이트리스트/basic-auth 1차 관문.
- 보안: 모든 `/api/admin/*`에 **서버 측 강제 → 403**. Django Ninja 라우터에 `auth=AdminAuth`(세션 인증 + `is_staff` 검증)를 전역 적용하고, 역할은 `require_role(...)` 의존성으로 검사(§15.5). 경로 은닉·프론트 가드는 **UX 보조일 뿐 보안 아님.**
- 세션: **일반 로그인으로 관리자 권한 획득 불가**(LoginScreen 관리자 분기 삭제). ⚠️ 단, `mgUser`/`mgAdmin`은 프론트 클라이언트 키일 뿐이고, 실제 서버 세션은 Django `sessionid` 쿠키 **하나**로 도메인 공유된다 — 이 부분은 레드팀 지적대로 서버 쿠키 격리가 별도로 필요하다(§15.1에서 보강).
- 계정: `create_initial_admins` 커맨드, 최초 로그인 강제 비번 변경, 로그인 실패 잠금(5회/10분) + nginx `limit_req`.
- 백엔드는 그린필드(`models.py`·`views.py`·`urls.py` 빈 스텁) — 신규 작성.

### 2.1 한자 콘텐츠 제외에 따른 영향

- **기존 `AdminContentScreen.tsx`(한자 목업)**: 한자 목록·검색·CRUD·출처 탭 UI는 **게시물 관리 화면(`AdminPostsScreen`)으로 대체**한다. 한자 관련 컴포넌트/목업(`admin.mock.ts`의 `ADMIN_HANJA_ROWS`·`ADMIN_SOURCE_ROWS`)은 사용하지 않으며, 재사용 가능한 공통 요소(테이블·페이지네이션·검색바·다이얼로그)만 게시물 화면으로 이관한다.
- 한자 데이터는 **추천 파이프라인(FastAPI/Chroma/Neo4j) 내부 자산으로 남기고**, 관리자 UI에서 편집하지 않는다. 따라서 정합성 동기화 이슈는 이번 범위에서 사라진다. 한자 수정이 필요하면 데이터팀 스크립트/Django 기본 어드민으로 처리(운영 저빈도 작업).
- 대시보드의 한자 관련 지표(있다면)는 게시물/문의/회원 지표로 교체한다.

## 4. 역할 등급 및 권한 매트릭스

### 4.1 역할 설계

| 역할 | 코드 | 설명 | Django 플래그 |
|------|------|------|----------------|
| 최고관리자 | `SUPERADMIN` | 관리자 계정/역할 관리, 시스템 설정, 백업/복구, 세션 강제 종료 등 전권 | `is_staff=True`, `is_superuser=True` |
| 운영관리자 | `ADMIN` | 일상 운영: 회원·게시물·문의·FAQ·공지·가입 승인. 관리자 계정 관리 불가 | `is_staff=True` |
| 분석/읽기전용 | `ANALYST` | 대시보드·통계·목록 **조회만**. 쓰기·상태변경·PII 상세 제한 | `is_staff=True` |

- 역할은 `AdminProfile.role`(TextChoices)로 관리. `is_superuser`는 `SUPERADMIN`에만 부여.
- 관리자 username은 이메일 형식이 아니므로 일반 가입(이메일=username)과 네임스페이스가 겹치지 않는다. 계정 생성 시 **`@` 포함 금지·예약 프리픽스(`admin`) 검증**으로 충돌·위장 가입을 원천 차단한다.
- **확장성**: 새 역할이 필요하면 `role` choices와 §4.2 매트릭스 행/열만 추가하면 되며, 코드 구조 변경 없음. 세분화가 커지면 Django `Group`/`Permission`으로 승격 가능(2차).

- 구현: Django Ninja 라우터에 `auth=AdminAuth`(is_staff)를 걸고, 엔드포인트별로 `require_role("SUPERADMIN")` 의존성을 추가해 역할을 검사. 매트릭스를 서버 상수(dict)로 두어 라우터·프론트가 같은 출처를 참조(프론트는 로그인 응답에 `role`+`permissions` 배열을 받아 UI 제어).

## 5. 개인정보(PII) 마스킹 설계

### 5.1 마스킹 규칙 (목록·검색 결과)

| 필드 | 원본 | 마스킹 | 규칙 |
|------|------|--------|------|
| 이메일 | `daesung@naver.com` | `dae***@naver.com` | 로컬파트 앞 3자 유지(3자 미만이면 1자+`*`), 나머지 `***`, 도메인 노출 |
| 이름 | `윤대성` | `윤*성` | 첫·끝 1자 유지 중간 `*`(2자면 `윤*`) |
| 전화(있으면) | `01012345678` | `010****5678` | 앞3·뒤4 유지 |

- 마스킹은 **서버가 직렬화 단계에서 수행**(프론트 마스킹은 원본이 이미 전송되므로 무의미·유출 위험). 목록 응답에는 마스킹 값만 담는다.

### 5.2 전체 노출 권한 기준

- **상세 화면에서만** 전체 PII 노출. 상세 조회 API가 **`role`에 따라** 전체/마스킹을 분기: `SUPERADMIN`·`ADMIN`은 전체(단 **`PiiAccessLog`에 조회 기록** — 누가·언제·어떤 회원을), `ANALYST`는 상세에서도 마스킹 유지.
- 목록 CSV/엑셀 내보내기는 **SUPERADMIN 전용** + 감사 기록. (기본값)
- 대안: 규제 민감도가 높으면 상세 전체 노출도 SUPERADMIN 전용으로 좁히고 ADMIN은 "노출 요청→사유 입력→일시 노출" 방식으로 강화(2차).

## 6. 데이터 분석·통계 설계

### 6.1 집계 방식 (기본값 + 대안)

- **기본값: 집계 테이블(`DailyMetric`) + 야간 배치.** 원천 로그(`LoginHistory`·`NamingRequest`·`ContactInquiry`)를 Django 관리 커맨드 `aggregate_daily_stats`가 **하루 1회 cron으로 집계**해 날짜별 1행 저장. 대시보드/통계는 이 테이블만 읽어 빠르고 부하 적음. 주간·월간은 일 단위 합산.
  - cron은 EC2 crontab 또는 docker-compose의 경량 스케줄러 사용(**메시지 큐 도입 안 함** — 제약 준수).
- **대안: 온디맨드 집계 + 캐시.** 트래픽이 작은 초기에는 요청 시 `NamingRequest` 등을 집계하고 결과를 짧게 캐시. 데이터 증가 시 기본값으로 승격.
- 원천 로그는 어차피 활동 로그(§9 모델)로 쌓이므로 **별도 분석 DB 불필요**(제약 준수).

### 6.2 차트 구성

- **트래픽 추이**: 라인 차트(일/주/월 토글) — 방문/작명요청/신규가입 다중 시리즈.
- **작명 요청 출처 분포**: 도넛(자연어 vs 상세조건 등).
- **전환 퍼널(요약)**: 랜딩→입력→결과→문의 단계 막대(간이). 정밀 퍼널은 2차.
- **문의 처리 현황**: 접수/처리 중/완료 스택 막대.
- 기간 필터(오늘/7일/30일/사용자 지정)를 통계·대시보드 공통 패턴으로 통일.

### 6.3 실시간 모니터링 · 사용자 행동 분석 (비용 대비 효과 권고안)

| 항목 | 효과 | 비용 | 권고 |
|------|------|------|------|
| 실시간 모니터링 대시보드(초 단위) | 장애 즉시 인지 | 폴링/스트리밍·서버 부하↑ | **2차 보류**. 1차는 §8 시스템 헬스 위젯(수동 새로고침/30초 폴링)으로 충분 |
| 사용자 행동 분석(퍼널·리텐션·세그먼트) | 제품 개선 인사이트 | 이벤트 수집·저장·분석 설계 필요 | **2차 보류**. 1차는 §6.2 요약 퍼널까지만. 본격 분석은 활동 로그 축적 후 |

→ 두 항목 모두 **팀 확정 전까지 2차**. 1차 집계·헬스로 운영 필요를 먼저 충족.

## 7. 회원가입 승인/거절 워크플로

- 흐름: **가입 → `PENDING`(로그인 제한/제한적 접근) → 관리자 승인 시 `APPROVED`(정상 이용) / 거절 시 `REJECTED`(사유 기록·안내)**.
- `UserProfile.approval_status` + `rejected_reason` + `approved_by` + `approved_at`.
- 승인 강제 여부는 **설정 플래그 `signup_requires_approval`로 on/off**(서비스 설정). 워크플로·모델·화면은 항상 준비하되 **강제 여부만 토글**.
  - **기본값**: 초기 오픈 부담을 줄이기 위해 `signup_requires_approval=false`(자동 승인)로 두되, 관리자 화면의 "가입 대기" 탭·승인/거절 기능은 구현해 두어 언제든 켤 수 있게 함.
  - **대안**: B2B/폐쇄형 운영이면 `true`로 시작.
- 거절/정지 시 이메일 안내는 §8 이메일 발송(2차)과 연동. 1차는 상태·사유 DB 기록까지.

## 9. 모델 스키마 (Django `webapp/naming/models.py` 초안)

> 그린필드이므로 아래를 신규 작성. 감사 필드(`created_at`/`updated_at`/`*_by`)를 공통 부여해 **확장 시 구조 변경 없이** 이력 추적이 가능하게 한다.

```python
from django.conf import settings
from django.db import models

USER = settings.AUTH_USER_MODEL


# ── 공통 추상 베이스 (확장성: 모든 운영 모델이 감사 필드를 상속) ──
class TimeStamped(models.Model):
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    class Meta:
        abstract = True


# ── 관리자 역할/프로필 ──
class AdminProfile(TimeStamped):
    class Role(models.TextChoices):
        SUPERADMIN = "SUPERADMIN", "최고관리자"
        ADMIN = "ADMIN", "운영관리자"
        ANALYST = "ANALYST", "분석/읽기전용"
    user = models.OneToOneField(USER, on_delete=models.CASCADE, related_name="admin_profile")
    role = models.CharField(max_length=20, choices=Role.choices, default=Role.ADMIN)
    display_name = models.CharField(max_length=50)
    is_active_admin = models.BooleanField(default=True)        # 관리자 정지
    must_change_password = models.BooleanField(default=True)
    failed_login_count = models.PositiveIntegerField(default=0)
    locked_until = models.DateTimeField(null=True, blank=True)
    created_by = models.ForeignKey(USER, null=True, blank=True,
                                   on_delete=models.SET_NULL, related_name="created_admins")


# ── 일반 회원 프로필 (가입 승인 워크플로) ──
class UserProfile(TimeStamped):
    class Approval(models.TextChoices):
        PENDING = "PENDING", "승인 대기"
        APPROVED = "APPROVED", "승인"
        REJECTED = "REJECTED", "거절"
    class Status(models.TextChoices):
        ACTIVE = "ACTIVE", "활성"
        SUSPENDED = "SUSPENDED", "정지"
        WITHDRAWN = "WITHDRAWN", "탈퇴"
    user = models.OneToOneField(USER, on_delete=models.CASCADE, related_name="profile")
    status = models.CharField(max_length=20, choices=Status.choices, default=Status.ACTIVE)
    approval_status = models.CharField(max_length=20, choices=Approval.choices,
                                       default=Approval.APPROVED)
    rejected_reason = models.CharField(max_length=200, blank=True)
    approved_by = models.ForeignKey(USER, null=True, blank=True,
                                    on_delete=models.SET_NULL, related_name="approved_users")
    approved_at = models.DateTimeField(null=True, blank=True)
    is_deleted = models.BooleanField(default=False)           # 소프트 삭제
    deleted_at = models.DateTimeField(null=True, blank=True)
    # 소프트 삭제 시 User.username(=email)에 접미사를 붙여 UNIQUE 회피(§15.3).
    # 원본 이메일은 재가입 판별/감사용으로 여기에 보존한다.
    original_email = models.EmailField(blank=True)


# ── 사용자 활동 로그 ──
class LoginHistory(models.Model):
    user = models.ForeignKey(USER, on_delete=models.CASCADE, related_name="login_history")
    ip = models.GenericIPAddressField(null=True, blank=True)
    user_agent = models.CharField(max_length=300, blank=True)
    success = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["user", "created_at"])]


class NamingRequest(TimeStamped):        # 작명 요청 이력(활동 로그 겸 통계 원천)
    class Kind(models.TextChoices):
        NATURAL = "natural", "자연어"
        STRUCTURED = "structured", "상세조건"
    user = models.ForeignKey(USER, null=True, blank=True,
                             on_delete=models.SET_NULL, related_name="naming_requests")
    kind = models.CharField(max_length=20, choices=Kind.choices)
    query = models.TextField(blank=True)
    result_count = models.PositiveIntegerField(default=0)
    status = models.CharField(max_length=20, default="done")
    class Meta:
        indexes = [models.Index(fields=["created_at"])]


# ── 관리자 감사 로그 (행위·PII 조회·권한 변경 통합) ──
class AdminAuditLog(models.Model):
    class Action(models.TextChoices):
        VIEW_PII = "VIEW_PII", "PII 상세 조회"
        ROLE_CHANGE = "ROLE_CHANGE", "권한 변경"
        USER_UPDATE = "USER_UPDATE", "회원 수정"
        USER_DELETE = "USER_DELETE", "회원 삭제"
        APPROVE = "APPROVE", "가입 승인/거절"
        CONTENT = "CONTENT", "콘텐츠 변경"
        LOGIN = "LOGIN", "관리자 로그인"
        OTHER = "OTHER", "기타"
    actor = models.ForeignKey(USER, null=True, on_delete=models.SET_NULL,
                              related_name="audit_logs")
    actor_username = models.CharField(max_length=150, blank=True)  # 스냅샷: 계정 삭제/변경돼도 주체 보존
    action = models.CharField(max_length=20, choices=Action.choices)
    target_type = models.CharField(max_length=50, blank=True)   # 예: "User", "Post"
    target_id = models.CharField(max_length=50, blank=True)
    detail = models.JSONField(default=dict, blank=True)         # before/after 등
    ip = models.GenericIPAddressField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["action", "created_at"]),
                   models.Index(fields=["actor", "created_at"])]


# PII 상세 조회 전용 뷰가 필요하면 AdminAuditLog(action=VIEW_PII)로 통일 → 별도 모델 불필요


# ── 게시물(콘텐츠 관리) ──
class PostCategory(TimeStamped):
    name = models.CharField(max_length=50, unique=True)
    order = models.PositiveIntegerField(default=0)


class Post(TimeStamped):
    class Status(models.TextChoices):
        PUBLISHED = "PUBLISHED", "게시"
        PRIVATE = "PRIVATE", "비공개"
        DELETED = "DELETED", "삭제"
    category = models.ForeignKey(PostCategory, on_delete=models.PROTECT, related_name="posts")
    title = models.CharField(max_length=200)
    body = models.TextField()
    status = models.CharField(max_length=20, choices=Status.choices, default=Status.PRIVATE)
    author = models.ForeignKey(USER, null=True, on_delete=models.SET_NULL, related_name="posts")
    updated_by = models.ForeignKey(USER, null=True, blank=True,
                                   on_delete=models.SET_NULL, related_name="updated_posts")
    published_at = models.DateTimeField(null=True, blank=True)


# ── 공지 ──
class Notice(TimeStamped):
    class Status(models.TextChoices):
        DRAFT = "DRAFT", "임시"
        SCHEDULED = "SCHEDULED", "예약"
        PUBLISHED = "PUBLISHED", "게시"
        ENDED = "ENDED", "종료"
    title = models.CharField(max_length=200)
    body = models.TextField()
    status = models.CharField(max_length=20, choices=Status.choices, default=Status.DRAFT)
    is_pinned = models.BooleanField(default=False)
    start_at = models.DateTimeField(null=True, blank=True)
    end_at = models.DateTimeField(null=True, blank=True)
    updated_by = models.ForeignKey(USER, null=True, blank=True, on_delete=models.SET_NULL)


# ── FAQ / 카테고리 ──
class FAQCategory(TimeStamped):
    name = models.CharField(max_length=50, unique=True)
    order = models.PositiveIntegerField(default=0)


class FAQ(TimeStamped):
    category = models.ForeignKey(FAQCategory, on_delete=models.PROTECT, related_name="faqs")
    question = models.CharField(max_length=200)
    answer = models.TextField()
    is_active = models.BooleanField(default=True)
    order = models.PositiveIntegerField(default=0)
    updated_by = models.ForeignKey(USER, null=True, blank=True, on_delete=models.SET_NULL)


# ── 문의 ──
class ContactInquiry(TimeStamped):
    class Status(models.TextChoices):
        RECEIVED = "received", "접수"
        IN_PROGRESS = "in_progress", "처리 중"
        ANSWERED = "answered", "답변 완료"
    name = models.CharField(max_length=50)
    email = models.EmailField()
    topic = models.CharField(max_length=50, blank=True)
    subject = models.CharField(max_length=200)
    message = models.TextField()
    status = models.CharField(max_length=20, choices=Status.choices, default=Status.RECEIVED)
    admin_reply = models.TextField(blank=True)
    answered_by = models.ForeignKey(USER, null=True, blank=True, on_delete=models.SET_NULL)
    answered_at = models.DateTimeField(null=True, blank=True)


# ── 통계 집계 테이블 (야간 배치) ──
class DailyMetric(models.Model):
    date = models.DateField(unique=True)
    signups = models.PositiveIntegerField(default=0)
    logins = models.PositiveIntegerField(default=0)
    naming_requests = models.PositiveIntegerField(default=0)
    inquiries = models.PositiveIntegerField(default=0)
    source_breakdown = models.JSONField(default=dict, blank=True)  # {"natural":n,"structured":m}
    class Meta:
        indexes = [models.Index(fields=["date"])]


# ── 서비스 설정 (운영 토글 최소) ──
class ServiceSetting(models.Model):
    key = models.CharField(max_length=50, unique=True)   # 예: "signup_requires_approval"
    value = models.JSONField(default=dict)
    updated_by = models.ForeignKey(USER, null=True, blank=True, on_delete=models.SET_NULL)
    updated_at = models.DateTimeField(auto_now=True)
```

설계 메모: `TimeStamped` 상속과 `AdminAuditLog`의 `target_type`/`target_id`/`detail(JSON)` 범용 구조 덕분에, **신규 도메인이 늘어도 감사·이력 체계를 그대로 재사용**한다(확장성 §11).

### 15.1 관리자 서버 세션 격리 (지적 1.1 — 수용)

지적이 정확하다. `mgUser`/`mgAdmin`은 프론트 클라이언트 키일 뿐이고, Django는 도메인당 **단일 `sessionid` 쿠키**를 쓰므로 `/`와 `/manage/`가 같은 쿠키를 공유한다. 같은 브라우저에서 사용자·관리자 로그인이 서로 덮어쓸 수 있다.

- **채택안(경량)**: 관리자 요청 경로에서만 **별도 쿠키 이름·경로**를 쓰는 세션 미들웨어. 요청 path가 `/api/admin/`(또는 `/manage/`)이면 `settings.SESSION_COOKIE_NAME='admin_sessionid'`, `SESSION_COOKIE_PATH='/api/admin/'`로 스왑하는 얇은 커스텀 `SessionMiddleware` 서브클래스. 이러면 사용자/관리자 세션이 **물리적으로 다른 쿠키**가 되어 상호 덮어쓰기·전이가 불가능하다.
- **보고서의 JWT 제안은 반려(기본값 기준).** 현재 스택이 세션+CSRF 쿠키 기반이라 JWT 전면 도입은 리팩터 비용·토큰 탈취/무효화 설계 부담이 크다. 쿠키명·경로 분리로 동일 목표(완전 격리)를 더 싸게 달성한다. JWT는 향후 모바일/외부 클라이언트가 생길 때 재검토(대안).
- **보강**: 관리자 세션 쿠키는 `HttpOnly`+`Secure`+`SameSite=Lax` 강제, 짧은 만료(예: 8h)와 idle timeout.

### 15.2 관리자 계정 하드 삭제 폐기 (지적 1.2 — 수용)

정확한 지적. 관리자를 하드 삭제하면 `AdminAuditLog.actor`(SET_NULL)가 `NULL`이 되어 악의적 관리자가 자기 흔적을 지울 수 있다.

- 관리자 계정(`User`+`AdminProfile`)은 **하드 삭제 금지, 정지(`is_active_admin=false`)만**. API에서 `DELETE /accounts/{id}` 제거(§10 반영).
- 추가 방어로 `AdminAuditLog.actor_username` **스냅샷 컬럼**을 두어(모델 반영), 설령 계정이 사라져도 행위 주체 문자열이 남게 한다.
- 자기 자신·마지막 활성 SUPERADMIN 정지도 금지(락아웃 방지, 기존 유지).

### 15.3 소프트 삭제 ↔ 이메일 UNIQUE 회피 (지적 1.3 — 수용)

정확한 지적. `User.username`(=email)은 UNIQUE라, 소프트 삭제된 회원의 이메일로 재가입하면 제약 위반이 난다.

- **채택안**: 소프트 삭제 시 `User.username`을 `deleted_{timestamp}_{원본}`으로 변경해 UNIQUE를 회피하고, 원본 이메일은 `UserProfile.original_email`에 보존(모델 반영). 재가입은 정상 허용, 이력은 추적 가능.
- **대안(그린필드 이점 활용)**: 지금 백엔드가 빈 스텁이므로 **커스텀 User 모델**을 도입해 이메일에 부분 UNIQUE 인덱스(`UniqueConstraint(condition=Q(is_deleted=False))`)를 거는 방법도 가능하다. 더 깔끔하지만 커스텀 User 전환 비용이 있어, 일정이 급하면 채택안(접미사)로 진행. **커스텀 User 도입 여부는 지금 결정하는 게 가장 싸다**(부록 확정 항목 추가).

### 15.4 계정 잠금 DoS 완화 (지적 2.2 — 수용)

타당하나, 본 설계는 이미 **`/manage/`·관리자 API에 nginx IP 화이트리스트**가 있어 공격자가 로그인 엔드포인트 자체에 도달하기 어렵다(1차 완화). 그 위에:

- 계정 기준 잠금 + **IP 기준 rate limiting**(nginx `limit_req`를 `/api/admin/login`에 적용) 병행.
- 반복 잠금 발생 시 **SUPERADMIN에 보안 알림**(이메일/슬랙 — §8 이메일 발송 2차와 연동).
- SUPERADMIN이 잠금을 즉시 해제하는 `POST /accounts/{id}/unlock`(§10 반영) — 정당한 관리자가 DoS로 영구 차단되지 않게 우회 경로 확보.

### 15.5 관리자 API 프레임워크: Django Ninja 채택 (지적 2.1 — 팀 결정 반영)

레드팀은 순수 FBV의 보일러플레이트(요청 파싱·검증·직렬화·에러 처리를 뷰마다 수작업)를 지적했고, 팀은 **FBV가 고정 요건이 아님**을 확인해 **Django Ninja로 확정**했다. 이 프로젝트 조건에 정확히 맞는 결정이다.

**채택 근거**
- 백엔드가 **빈 스텁(그린필드)** 이라 전환 비용이 지금 거의 0. (나중에 옮기면 비쌈 → 지금이 최적기)
- 팀이 **이미 FastAPI로 Pydantic을 사용** 중이라 Ninja(Pydantic 기반)의 학습곡선이 낮음 — "Django 위의 FastAPI"에 가깝다.
- `Schema` 기반 **검증·직렬화 자동화** + **OpenAPI 문서 자동 생성**으로 관리자 API 40~50개의 반복 노동·휴먼에러를 제거.
- DRF는 이 규모에 무겁고, 마스킹·감사 등 커스텀이 많은 관리자 도메인에선 뷰셋 커스터마이징이 오히려 번거로워 제외.

**설계 규약 (Ninja 기준)**
- 라우터 구성: `webapp/naming/api.py`에 `NinjaAPI(urls_namespace="admin", docs_url="/docs", auth=AdminAuth())`를 두고, 도메인별 `Router`(users/posts/inquiries/faqs/notices/accounts/audit/stats)를 `add_router`로 조립.
- 인증/인가: `AdminAuth`(Ninja `HttpBearer`/세션 커스텀 auth)가 세션 검증 + `is_staff`를 강제(비로그인 401, 비관리자 403). 역할은 `require_role("SUPERADMIN")` 의존성으로 엔드포인트별 검사. 권한 매트릭스(§4.2)는 서버 상수로 두고 `/me`가 프론트에 권한 배열 반환.
- 스키마: 입력 `XxxIn(Schema)` / 출력 `XxxOut(Schema)`. **마스킹은 출력 스키마의 `resolve_*` 또는 역할별 스키마 분기**로 처리(§5). Pydantic 스키마는 프론트 Zod(§16.2)와 필드를 정렬해 계약 일치.
- 페이지네이션: `django-ninja`의 `paginate` + 커스텀 `PageNumberPagination`으로 `{items,total,page,pageSize}` 통일. 로그성 목록은 커서 페이지네이션(§15.7).
- 예외/에러 포맷: `@api.exception_handler`로 검증·중복(409)·권한(403)을 `{status,message,detail}`(client.ts `ApiError` 호환)로 통일.
- 기존 사용자용 Django 뷰(인증·마이페이지 등)는 그대로 두고 **관리자 API만 Ninja 라우터로 신규 작성**(그린필드라 충돌 없음). 필요 시 사용자 API도 점진 이관 가능(선택).

**대표 예시 (회원 생성)**
```python
class UserCreateIn(Schema):
    email: EmailStr
    name: str

class UserOut(Schema):
    id: int; email: str; name: str; status: str

@router.post("/users", response={201: UserOut})
def create_user(request, payload: UserCreateIn, _=Depends(require_role("ADMIN"))):
    if User.objects.filter(username=payload.email).exists():
        raise HttpError(409, "중복 이메일")
    user = User.objects.create_user(username=payload.email, email=payload.email)
    UserProfile.objects.create(user=user)
    return 201, user
```

**의존성**: `django-ninja`(+ `pydantic`은 전이 의존). `requirements.txt`에 추가.

### 15.6 배치 멱등성·타임존 (지적 2.3 — 부분 수용)

- **타임존**: 이미 `settings.py`에 `USE_TZ=True`, `TIME_ZONE='Asia/Seoul'`가 설정돼 Django가 UTC 저장·KST 변환을 처리한다. 배치가 `timezone.localdate()`(KST 경계) 기준으로 집계하면 문제없다 — 구현 규칙으로 명시(naive 날짜 연산 금지).
- **멱등성(수용)**: 배치는 `update_or_create(date=...)` **upsert**로 동작해 재실행해도 중복 집계되지 않게 한다. 또한 지연 반영(늦게 처리된 문의 등) 대비 **최근 N일(예: 3일) 롤링 재집계**.
- **지적의 "상태 변경 통계" 프레이밍은 일부 반박**: `DailyMetric`은 `created_at` 기준 **이벤트 카운트**(가입·요청·문의 접수)로, 시점 고정값이라 재집계에 흔들리지 않는다. "현재 상태 분포(접수/처리중/완료)"는 배치가 아니라 **실시간 쿼리**로 보여주는 지표이므로 created_at 배치에 섞지 않는다(설계 명확화).

### 15.7 페이지네이션 전략 (지적 2.4 — 부분 수용)

- 회원·문의·게시물 목록은 관리 UI 특성상 **페이지 번호(Offset)** UX가 유리하고 규모도 상대적으로 유계 → Offset 유지 + **인덱스 + 최대 offset 가드**(예: page 상한, 초과 시 검색 조건 강제).
- 반면 **빠르게 무한 증가하는 로그성 목록**(`LoginHistory`·`NamingRequest`·`AdminAuditLog`)은 **커서(Keyset) 페이지네이션**을 채택(`created_at`+`id` 커서). 여기가 실제로 깊은 offset이 위험한 지점이다.

## 16. 프론트엔드 스택·배포 요구 검토 (2차 요구사항)

추가 요구 4건을 실제 코드(`frontend/package.json`, `src/app/providers/QueryProvider.tsx`, `src/app/components/ui/*`)와 대조해 판정했다.

| # | 요구 | 판정 | 근거 |
|---|------|------|------|
| 1 | 관리자 세션 쿠키 분리(`admin_sessionid`) | **수용(기반영)** | §15.1과 동일 사안 — 이미 채택. §16.1에서 백엔드 설정으로 명문화 |
| 2 | React Query + RHF + Zod 추가 | **부분 수용** | React Query는 **이미 스택에 존재·사용 중**(추가 아님). RHF+Zod만 신규 채택 |
| 3 | 정적 자산 경로 라우팅 에러 방지 | **수용** | MPA + `/manage/` 폴백의 실제 함정 — §16.3에서 상세화 |
| 4 | Tailwind v4 → v3.4 하향 | **반려** | 이 레포는 **이미 v4에서 shadcn식 컴포넌트 정상 동작 중** — 전제가 반증됨 |

### 16.1 관리자 세션 쿠키 분리 (요구 1 — 수용, §15.1 재확인)

이미 §15.1에서 채택한 사안이다. 백엔드 보안 설정에 다음을 **명문화**한다: 요청 path가 `/api/admin/`이면 세션 쿠키명을 `admin_sessionid`, 경로를 `/api/admin/`으로 발급하는 경량 세션 미들웨어. 사용자 `sessionid`와 물리적으로 분리되어 상호 덮어쓰기가 불가능하다. (프론트 계획서 항목이지만 실제 구현은 Django 설정 — 백엔드 담당.)

### 16.2 상태·폼 라이브러리 (요구 2 — 부분 수용)

- **React Query(@tanstack/react-query 5.62.0)는 이미 도입되어 있다.** `QueryProvider`가 앱을 감싸고, `useAdminUsers`·`useAdminDashboard`·`useFaq` 등 8개 훅이 `useQuery`로 서버 상태를 관리 중이다. 따라서 "추가"가 아니라 **관리자 CRUD로 확장**이 정확한 표현이다.
  - 현재는 조회(`useQuery`)만 쓰고 **`useMutation`은 아직 미사용**. 관리자 CRUD·낙관적 업데이트·`invalidateQueries` 캐시 무효화 패턴을 이번에 표준화한다(목록/상세 쿼리 키 컨벤션 포함).
- **React Hook Form + Zod는 신규 채택(수용).** 회원 수정·게시물·FAQ·공지·계정 등 폼이 많아 순수 `useState` 검증은 보일러플레이트·버그를 유발한다. RHF(비제어 폼 성능) + Zod(스키마 검증, 서버 계약과 타입 공유) 조합을 폼 표준으로 확정.
  - 신규 의존성: `react-hook-form`, `zod`, `@hookform/resolvers`. 경량이고 스택 원칙("현재 스택 내 해결")에 부합.
  - Zod 스키마는 `admin.ts` API 계약과 한 곳에서 공유해 프론트 검증과 서버 요청 타입을 일치시킨다(확장성).

### 16.3 정적 자산(Assets) 경로 라우팅 (요구 3 — 수용)

MPA로 `admin.html`을 추가하면 이 파일이 참조하는 `/assets/*.js|css`가 nginx `/manage/` 폴백과 어긋나 **빈 화면(White Screen)** 이 날 수 있다. 실제 함정이므로 배포 규칙을 확정한다.

- **Vite `base` 처리(채택안)**: 관리자 번들 산출물이 `/manage/` 하위 경로를 참조하도록, 관리자 빌드에 `base: '/manage/'`를 적용(사용자 번들은 `base: '/'` 유지). 그러면 `admin.html`은 `/manage/assets/...`를 요청한다.
- **nginx 매핑(둘 다 명시)**:
```nginx
# 관리자 정적 자산: 폴백보다 먼저 매칭되도록 별도 location
location /manage/assets/ {
    alias /usr/share/nginx/html/manage/assets/;   # Vite base=/manage/ 산출물 위치
    access_log off;
    expires 1y;
}
# 관리자 SPA 진입/폴백
location /manage/ {
    # (관문: IP 화이트리스트/basic-auth — §2·§15.4)
    root /usr/share/nginx/html;
    try_files $uri /manage/admin.html;            # 실경로 없으면 admin.html
}
```
- **핵심 원칙**: 정적 파일 요청은 **폴백(`try_files ... admin.html`)에 흡수되기 전에** 실제 파일로 매칭되어야 한다. `location /manage/assets/`를 `/manage/` 폴백보다 구체적으로 두어 우선 매칭시킨다. 빌드 산출물의 자산 물리 경로(`html/manage/assets/`)와 `alias`가 일치하는지 배포 검증에 포함(§14 R6).
- **대안**: `base`를 바꾸지 않고 사용자·관리자가 `/assets/`를 공유하되, `admin.html`만 `/manage/`에서 서빙하고 `/assets/`는 `location /`이 그대로 서빙. 이 경우 `try_files`가 `/assets/` 실파일을 먼저 찾으므로 동작하지만, 경로 은닉·캐시 정책 분리 측면에서 채택안(`base:/manage/`)이 더 깔끔하다.

### 16.4 Tailwind 버전 (요구 4 — 반려, 근거 제시)

**하향을 반려한다. 요구의 전제가 이 레포에서 반증되기 때문이다.**

- 현재 `tailwindcss 4.1.12` + `@tailwindcss/vite`를 쓰면서 **이미 shadcn/ui 방식 컴포넌트(Radix accordion·alert-dialog·dialog·dropdown-menu·tabs·switch + CVA·tailwind-merge)가 정상 동작 중**이다(`src/app/components/ui/*`). "v4 `@theme`가 shadcn과 구조적 비호환"이라는 주장은 실제 구동 사실과 배치된다. shadcn/ui는 2025년부터 Tailwind v4를 공식 지원한다.
- v4→v3.4 하향은 **전체 앱**의 `theme.css`·`tailwind.css`·PostCSS·`@theme` 토큰을 재작성해야 하는 **대규모 회귀**다. 이미 v4에서 완성된 사용자 화면 스타일까지 흔들어 위험 대비 실익이 없다.
- **채택안**: v4 유지. 관리자 화면도 기존 `components/ui/*`를 재사용(§12 UX 지침과 일치). 특정 신규 컴포넌트가 v4에서 문제가 나면 **그 컴포넌트만 국소 대응**(shadcn v4 버전 사용 또는 수동 조정)한다. 전면 하향은 하지 않는다.
- (만약 팀이 v3 전용 서드파티 컴포넌트를 반드시 써야 한다면, 그건 스택 전체 결정이므로 별도 안건으로 다루되 기본값은 v4 유지.)

## 14. 검증 계획

### 서버 RBAC·권한 매트릭스 (필수)
1. 비로그인 `/api/admin/*` → **401**.
2. 일반 사용자 세션 → 모든 `/api/admin/*` **403**(프론트/nginx 우회 시나리오).
3. `ANALYST`가 쓰기 API(회원 수정·게시물 CRUD 등) → **403**.
4. `ADMIN`이 `/api/admin/accounts*`·`/audit-logs`·`/settings` → **403**.
5. `SUPERADMIN`만 역할 변경 가능, 변경 시 `AdminAuditLog(action=ROLE_CHANGE)` 기록 확인.

### PII 마스킹
6. 목록 응답에 마스킹 값만 포함(원본 이메일/이름 미포함) — 응답 바디 문자열 검사.
7. `ADMIN`이 상세 조회 시 전체 PII + `AdminAuditLog(action=VIEW_PII)` 1건 생성.
8. `ANALYST` 상세 조회는 마스킹 유지, 내보내기 API 접근 403.

### 가입 승인 워크플로
9. `signup_requires_approval=true`에서 신규 가입 → `PENDING`, 로그인/보호 기능 접근 차단.
10. 관리자 승인 → `APPROVED` 전이, 로그인 가능. 거절 → `REJECTED` + 사유 저장.

### 회원 관리·자기보호
11. 관리자가 자기 자신 정지/삭제 → **400**. 마지막 활성 SUPERADMIN 삭제 → **400**.
12. 소프트 삭제 후 목록 기본 필터에서 제외되나 복구 가능.

### 무결성·세션 (레드팀 반영 §15)
R1. **세션 격리**: 같은 브라우저에서 사용자 로그인 후 `/manage` 관리자 로그인 → 두 세션 쿠키(`sessionid`/`admin_sessionid`) 공존, 어느 쪽도 상대를 로그아웃시키지 않음.
R2. **관리자 삭제 불가**: `DELETE /api/admin/accounts/{id}` 미존재(404/405), 정지만 가능. 정지된 관리자의 과거 감사 로그 `actor_username` 유지.
R3. **재가입**: 회원 소프트 삭제 후 동일 이메일 재가입 성공(UNIQUE 위반 없음), `original_email`에 원본 보존.
R4. **잠금 DoS/해제**: 5회 실패 잠금 후 SUPERADMIN `unlock`으로 즉시 해제, 반복 잠금 시 알림 트리거.
R5. **배치 멱등성**: `aggregate_daily_stats` 2회 연속 실행해도 `DailyMetric` 값 불변(중복 집계 없음), 3일 롤링 재집계로 지연 데이터 반영.
R6. **관리자 정적 자산**: 배포 후 `/manage/`(및 하위 딥링크) 진입 시 흰 화면 없이 렌더, DevTools Network에서 `/manage/assets/*` 200 응답(폴백에 흡수돼 admin.html/HTML로 반환되지 않음). 관리자 폼에서 RHF+Zod 검증 동작 확인.
R7. **Ninja 스키마·문서**: 잘못된 입력(형식 위반) 전송 시 Ninja가 표준 422/400 검증 오류 반환, `/api/admin/docs` OpenAPI 문서가 관리자 인증 뒤에서만 열림(비인증 접근 차단).

### 통계·집계
13. `aggregate_daily_stats` 실행 후 `DailyMetric` 해당 날짜 1행, 주간/월간 합산 값이 원천 로그 카운트와 일치.

### 라우팅·분리 (전신 문서 승계 확인)
14. `/faq` 새로고침 200, `/manage/*` 딥링크 폴백 정상, 사용자 번들에 admin 코드 0바이트(`grep` 검증).

### 프론트(권한 UI)
15. `/api/admin/me` 권한 배열에 따라 메뉴/버튼 노출이 매트릭스와 일치(서버 강제와 이중 확인).

### 자동화
```bash
cd webapp
python manage.py create_initial_admins
python manage.py aggregate_daily_stats     # 집계 배치 스모크
python manage.py test naming.tests         # 위 1~15 + R1~R5 케이스
cd ../frontend && npm run build && ! grep -rl "AdminLayout\|AdminDashboardScreen" dist/assets/*.js
```
