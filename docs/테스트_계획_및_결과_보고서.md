## 4. LLM API 연동 테스트

### 4.1 비동기 예외 처리 구현

`fastapi_app/main.py`의 `POST /names/generate`는 LangGraph 파이프라인 호출을 다음과 같이 예외별로 구분해 처리한다.

```python
try:
    results = await asyncio.wait_for(_generate_structured(query, exclude_names), timeout=90)
except asyncio.TimeoutError:
    return JSONResponse(status_code=504, content={"message": "작명 생성 시간이 초과되었습니다.", "detail": None})
except NeedMoreInfoError as exc:
    return JSONResponse(status_code=422, content={"message": "이름을 생성하려면 추가 정보가 필요합니다.", "detail": exc.guidance})
except Exception as exc:
    return JSONResponse(status_code=502, content={"message": "작명 생성에 실패했습니다.", "detail": str(exc)})
```

### 4.2 실제 검증 결과

| # | 시나리오 | 방법 | 결과 |
|---|---|---|---|
| 1 | 정상 호출(개수 미지정) | `POST /naming-api/names/generate` (자연어 쿼리, 개수 미지정) | 200, 이름 5개 반환 (2026-07-09, a6d7ad9 배포 후 프로덕션 실측) |
| 2 | 정보 부족 시 반문(clarify) | LLM이 이름 대신 반문(`## [이름` 헤더 없음)을 반환하는 케이스 | `NeedMoreInfoError` → 422, `detail`에 반문 안내 전달 |
| 3 | LLM 응답 시간 초과 | 90초 초과 시나리오 | `asyncio.TimeoutError` → 504 |
| 4 | 외부 API 키 소진(429) | 실제 운영 중 OpenAI 크레딧 소진 발생 | `insufficient_quota` 429가 502 응답의 `detail`로 그대로 노출됨을 확인. 팀원 키로 `.env` 교체 후 `docker compose up -d --force-recreate fastapi`로 정상화(단순 `restart`는 `.env` 재적재 안 됨을 확인) |
| 5 | 컨테이너 기동 직후 일시적 502 | fastapi 컨테이너 재기동 직후 임베딩 로딩 중 nginx 프록시 요청 | 기동 완료(`Application startup complete.`) 전까지 502 발생 → 완료 후 정상화 확인(일시적 현상, 재시도로 해소) |

### 4.3 오행(五行) 조건 필터링 정합성 테스트

## 5. 배포 환경 검증

### 5.1 컨테이너 재기동·재현성

| 항목 | 검증 내용 |
|---|---|
| `restart: unless-stopped` | 전 서비스(nginx/django/fastapi/neo4j)에 적용(커밋 `55c46aa`). EC2 인스턴스 재부팅 시 수동 개입 없이 자동 기동 |
| Docker 데몬 다운 복구 | 로컬 개발 중 Docker Desktop 데몬이 다운된 사례에서, 데몬 재기동만으로 `restart: unless-stopped` 정책에 따라 기존 컨테이너들이 자동으로 재기동됨을 확인 |
| 이미지 재빌드 재현성 | `docker compose up --build -d`로 코드 변경분을 반영한 재빌드가 매번 동일하게 동작함을 세션 전반의 배포마다(총 10회 이상) 반복 확인 |
| `.env` 변경 반영 | `docker compose restart`는 `env_file`을 재적재하지 않음을 실제 장애(OpenAI 키 교체)로 확인 — `up -d --force-recreate <service>`가 필요함을 검증하고 문서화 |

### 5.3 마이그레이션 안정성 검증 (daesung 브랜치 병합 사례)

daesung 브랜치(관리자 페이지 기능) 병합 시, 별도 테스트 브랜치(`_test_merge_daesung`)에서 사전 위험 분석을 수행했다.

| 위험 | 내용 | 검증/조치 |
|---|---|---|
| 마이그레이션 파일명 충돌 | daesung이 이미 프로덕션에 적용된 `0001_initial.py`를 동일 파일명으로 전면 재작성하고 `0002_userconsent.py`를 `0002_trendarticle_nametrendstat.py`로 대체함. Django는 마이그레이션을 (앱, 파일명) 기준으로 적용 여부를 판단하므로, 그대로 병합 시 신규 테이블 8개가 "이미 적용됨"으로 오판되어 생성되지 않는 문제였음 | main의 `0001_initial`·`0002_userconsent`를 원본 그대로 복원하고, daesung의 신규 모델만 반영하는 `0003_admin_features_and_userprofile.py`를 `makemigrations`로 새로 생성. `UserConsent → UserProfile` 데이터 백필(`RunPython`)을 포함해 기존 가입자 동의 기록 보존 |
| ChromaDB 데이터 손실 위험 | daesung 브랜치에서 `data/chroma/` 40개 파일이 실수로 git에서 삭제 추적됨. 그대로 병합했다면 EC2 배포 시 `git pull` 한 번으로 운영 RAG 벡터 데이터 전체가 삭제될 뻔했음 | `git checkout origin/main -- data/chroma/` + `git add -f`로 전량 복원 확인 후 병합 |
| 중복 함수 정의 | `heartbeat_view`가 동일 이름으로 두 번 정의되어 파이썬이 뒤 정의로 덮어씀 — 프론트가 GET(점검 확인)·POST(세션 핑) 모두 호출하는데 POST가 항상 405 | 하나의 뷰로 병합해 GET/POST 모두 처리하도록 수정 |
