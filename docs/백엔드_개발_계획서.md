# 백엔드 개발 계획서 (명가작명소)

작성일: 2026-07-06
기준 코드: `frontend/`(React+Vite, mock 어댑터), `webapp/`(Django 스켈레톤), `fastapi_app/`(작명 QA API), `src/`(LangGraph·MCP 툴)
관련 문서: `docs/프론트-백엔드_통합_최종가이드.md`(도커·nginx·병합 절차), `submission/화면설계서_최종.md`

## 개요

프론트엔드 데모(목업 데이터, `USE_MOCK` 어댑터 패턴)가 완성된 상태다. 프론트 `src/api/*.ts`에는 이미 실 API 어댑터(realAdapter)와 엔드포인트 경로가 정의되어 있어, 백엔드는 **이 계약(Contract)을 그대로 구현**하면 `VITE_USE_MOCK=false` 전환만으로 연동된다.

백엔드는 두 서버로 분리되어 있다.

- **Django (`/api`, 8000)** — 인증·마이페이지·작명 기록·인사이트·고객센터·관리자 API. PostgreSQL 사용.
- **FastAPI (`/naming-api`, 8001)** — 작명 생성·채팅. LangGraph(ReAct) + ChromaDB + Neo4j + 외부 API(법령·우리말샘).

현재 상태 요약:

| 영역 | 현재 상태 | 갭 |
| --- | --- | --- |
| Django | `config` + `naming` 앱 스켈레톤만 존재 (models/views/urls 전부 비어 있음) | 모든 API·모델 신규 구현 필요 |
| FastAPI | `/health`, `/ask`(QA), `/graph/ohaeng`만 존재 | 프론트가 기대하는 `/names/generate`, `/names/sample-preview`, `/chat` 미구현 |
| 인프라 | docker-compose(postgres·nginx·django·fastapi·neo4j) 구성 완료 | nginx multi-stage 빌드 등은 통합가이드 STEP 2 참조 |

## 목표

- `VITE_USE_MOCK=false`로 전환 시 **전 화면이 목업 없이 동작** (수용 기준: 통합가이드 §10 체크리스트 전부 통과)
- 프론트 `types.ts`의 DTO와 백엔드 응답 스키마 **100% 일치** (계약 테스트로 검증)
- 작명 생성 API 응답 p95 30초 이내, 그 외 Django API p95 300ms 이내
- 인증·권한: 세션 기반 로그인, 관리자 API는 서버측 RBAC(is_staff)로 401/403 구분 응답
- 모든 오류 응답이 공통 JSON 포맷(`{status, message, detail}`)을 따름 — 프론트 `ApiError`와 일치

## 범위

- 포함:
  - Django REST API 전체(인증, 마이페이지, 기록, 인사이트, 고객센터, 관리자)
  - FastAPI 작명 생성·채팅 API (기존 LangGraph 재사용)
  - DB 모델링·마이그레이션, 작명 기록 저장 흐름
  - CSRF/세션 처리, 오류 포맷 표준화, 도메인별 USE_MOCK 점진 전환
- 제외 (이번 단계에서 하지 않음):
  - 소셜 로그인(OAuth), 이메일 실발송(비밀번호 재설정은 접수만 기록)
  - 결제·구독, 푸시 알림
  - 채팅 SSE 스트리밍(1차는 일반 POST 응답, 스트리밍은 후순위)
  - 인사이트 실데이터 수집 파이프라인(통계청 연동) — 1차는 시드 데이터를 DB에 적재해 서빙만 실API화

## 아키텍처 결정 사항

### A. Django는 DRF(Django REST Framework) 도입
serializer(스키마 검증), permission(RBAC), pagination을 직접 구현하지 않기 위해 DRF를 사용한다. `requirements.txt`에 `djangorestframework` 추가.

### B. 인증 방식: Django 세션 + CSRF
프론트 `client.ts`가 이미 `credentials: "include"` 로 구현되어 있고 단일 도메인(nginx) 구성이므로 세션 인증이 가장 단순하다. JWT는 도입하지 않는다.
- 필요 추가 작업: 프론트 `client.ts`에 CSRF 토큰 헤더(`X-CSRFToken`) 처리 — 통합가이드 §6.2 절차 그대로.
- 세션 복원용 `GET /api/auth/me` 엔드포인트가 필요하다(새로고침 시 로그인 유지). **프론트 realAdapter에 아직 없으므로 추가 계약** — AuthProvider 수정 필요. `[확인 필요: 프론트 담당과 협의]`

### C. 작명 기록 저장 흐름 (중요 결정)
FastAPI는 세션을 모르므로, 기록(History) 저장을 누가 하느냐를 정해야 한다.

| 옵션 | 방식 | 장점 | 단점 |
| --- | --- | --- | --- |
| 1 (권장) | 프론트가 generate 성공 후 `POST /api/me/history`로 결과 요약 저장 | 서버 간 결합 없음, 구현 단순 | 프론트 호출 2회, 유실 가능성(브라우저 이탈) |
| 2 | Django가 FastAPI를 서버측 프록시(`/api/names/generate` → 내부 호출) | 세션·기록 일원화, FastAPI 비공개화 가능 | Django 워커가 LLM 응답(수십 초) 동안 점유됨 |
| 3 | FastAPI가 Django 내부 API로 기록 push | 프론트 단순 | 서버 간 인증 토큰 관리 필요 |

데모 규모에서는 **옵션 1**을 권장. 트래픽·정합성 요구가 커지면 옵션 2로 이전.

### D. 오류 응답 표준
모든 4xx/5xx는 아래 JSON을 반환한다 (DRF exception handler + FastAPI exception handler 커스텀).

```json
{ "status": 403, "message": "권한이 없습니다.", "detail": { "code": "forbidden" } }
```

## 오류 예방 원칙 (개발 시 반드시 지킬 것)

- **계약 우선**: 필드명 하나라도 프론트 `types.ts`와 다르면 화면이 조용히 깨진다. camelCase 그대로 반환(DRF는 `djangorestframework-camel-case` 사용 또는 serializer 필드명 직접 지정).
- **204 규약**: 프론트 client는 204를 `undefined`로 처리한다. 본문 없는 성공은 반드시 204로.
- **401 처리**: 401 수신 시 프론트가 `unauthorizedHandler`로 로그인 리다이렉트한다. 로그인 실패(401)와 세션 만료(401)가 같은 코드이므로, 로그인 엔드포인트만은 리다이렉트 예외 처리 필요 — 프론트와 합의.
- **트랜잭션**: 회원 탈퇴, 기록+결과 동시 저장은 `transaction.atomic`으로 묶는다.
- **타임존**: `TIME_ZONE='Asia/Seoul'`, `USE_TZ=True` 통일. 프론트 표기 형식(`YYYY.MM.DD`)은 프론트에서 포맷.
- **LLM 호출 방어**: 타임아웃(60s)·재시도 1회·실패 시 표준 오류 반환. FastAPI 워커에서 동기 블로킹 금지(async 유지).
- **시크릿 관리**: `.env.example`에 **실제 OpenAI 키가 커밋되어 있음 — 즉시 키 폐기(rotate)하고 플레이스홀더로 교체할 것.** SECRET_KEY 기본값도 운영에서 반드시 env로 주입, `ALLOWED_HOSTS='*'` 제거.
- **개인정보**: 관리자 화면·로그의 이메일은 서버측에서 마스킹(`kim***@naver.com`)해 내려준다.

## 트러블슈팅 (증상 → 원인 → 조치)

| 증상 | 원인 | 조치 |
| --- | --- | --- |
| POST에서 403 "CSRF Failed" | CSRF 토큰 미전송 | `GET /auth/csrf` 선호출, `X-CSRFToken` 헤더 추가, `CSRF_TRUSTED_ORIGINS` 설정 |
| 로그인 후 새로고침하면 로그아웃 | 세션 복원 API 부재/미호출 | `GET /auth/me` 구현 + AuthProvider 초기 호출 |
| 로그인 되는데 이후 요청 401 | 쿠키 미포함/미저장 | fetch `credentials:"include"` 확인, 개발 시 vite proxy 경유(포트 직접 호출 금지), `SESSION_COOKIE_SAMESITE='Lax'` |
| 응답은 오는데 화면 값이 비어 있음 | 필드명 불일치(snake_case) | camelCase 직렬화 확인, 브라우저 Network 탭에서 실제 응답과 types.ts 대조 |
| `/api/...` 404 | nginx/vite proxy가 prefix를 잘못 strip | Django urls는 `/api/` 포함 여부를 proxy 설정과 맞춤 (통합가이드 §C) |
| 작명 요청이 504/timeout | LLM 응답 지연 > nginx 기본 60s | nginx `proxy_read_timeout 120s`, 프론트 타임아웃도 상향, Processing 화면 유지 |
| FastAPI 기동 실패 (fastmcp import) | `src/mcp` 경로·스텁 문제 | `fastapi_app/main.py`의 `_stub_fastmcp` 패턴 유지, PYTHONPATH 확인 |
| Neo4j 연결 오류 | 컨테이너 기동 순서/인증 | `NEO4J_URI=bolt://neo4j:7687`(포트 주의 — `.env.example`의 `bolt://neo4j:5`는 오기), healthcheck 후 기동 |
| Chroma 검색 결과 0건 | 볼륨 미마운트/인덱스 미생성 | `./data:/app/data` 마운트 확인, 인덱싱 스크립트 선실행 |
| migrate 충돌 | 팀원 간 마이그레이션 동시 생성 | 마이그레이션 담당 1인화, 충돌 시 `--merge` 대신 재생성 협의 |
| 개발 중 CORS 오류 | proxy 미경유 직접 호출 | 항상 vite dev 서버(5173) 경유, 운영은 nginx 단일 도메인이라 CORS 무관 |

그 외 도커·병합·배포 단계 오류는 `docs/프론트-백엔드_통합_최종가이드.md` §9 트러블슈팅에 이미 정리되어 있으므로 중복 기재하지 않는다.

## 가정

- 프론트 realAdapter의 엔드포인트 경로·DTO를 계약의 기준으로 삼았다(코드에서 직접 확인).
- 일정·담당은 미정으로 두었고 `[확인 필요]`로 표시했다.
