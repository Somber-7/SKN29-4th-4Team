### 점검에서 확인된 사실 (실측)

- 성씨 사전(`surname_ohaeng.json`) 163개 성씨 중 **65개(40%)가 동음이의** — 프론트 폼은 한글 성씨만 보내므로 전부 clarify로 빠졌다.
- 오행 정규식 `([木火土金水])오행`은 공백("木 오행")을 매칭하지 못했다. `naming_graph.py` 자체 테스트 케이스조차 매칭 실패 상태였다.
- clarify 반문 텍스트가 구조화 LLM에 "완성된 추천 결과"로 전달되어 빈 배열/지어낸 이름이 반환될 수 있었다.
- "김씨 8획"의 **성씨 획수가 이름 글자 획수 필터로 오적용**되는 누수 (3차 평가 Case 1 최저점 원인 후보).
- 구조화 변환 LLM(temperature 0.3)이 마크다운에 없는 성씨 획수/오행/뜻을 지어서 채우고 있었다.

## 4. 팀 결정 사항과 반영 (2026-07-07 결정 → 당일 구현)

| 항목 | 결정 | 상태 |

| 4-2 strokeRange | 추천 방향 제시 요청 → **이름 글자 획수 합(원격) 범위**로 정의하고 구현 | ✅ 구현 |

### 4-1. OR 오행 판정 구현

- **발음오행 계산**: 초성→오행 표 (`ㄱㅋ=목, ㄴㄷㄹㅌ=화, ㅇㅎ=토, ㅅㅈㅊ=금, ㅁㅂㅍ=수`). 이 유파 선택은 임의가 아니라 **데이터 실측** — hanja_col의 `sound_ohaeng` 필드(413개 한글 음, 충돌 0건)와 완전 일치하는 유파다.
- **기준 음절**: 성씨·이름 모두 실제 표기 음절로 계산 (DB 한자 독음은 원음(林=림)이라 표기(임)와 초성이 다름 — 표기 기준이 사용자 인지와 일치).
- **판정**: 두 기준 각각 체인 판정. 판정 가능한 기준이 **모두 상극일 때만 탈락**. 성씨 자원오행 미상이면 발음오행 단독으로 판정 가능(추천 가능 범위도 넓어짐).
- **실측 효과**: 임씨(林) 테스트에서 후보 47건 중 9건(19%)이 자원오행 단독 기준이라면 탈락했을 물량 — OR 규칙으로 확보.
- **표기**: 두 기준 모두 출력하고 통과 기준에 ✔ 표시:
  ```
  **오행 흐름(자원오행)**: 임씨(목) → 첫째(화) → 둘째(토) — 상생 ✔
  **오행 흐름(발음오행)**: 임(토) → 소(금) → 연(토) — 중립 ✔
  ```

### 4-2. strokeRange 권고안 (채택·구현): 이름 글자 획수 합(원격) 범위

근거: ① 폼 placeholder "예: 20~25" — 글자당 20~25획 한자는 희소해서 합산 해석이 자연스러움 ② 원격(元格)은 수리 4격의 기본 격이라 "획수를 지정해 수리를 조정한다"는 사용자 의도와 일치 ③ 외자는 글자 획수 자체로 해석. 글자당 범위로 바꾸려면 `_extract_stroke_range` 주석과 열거 필터의 비교 한 줄만 수정하면 된다.

### 4-3. 다중 오행 반영

`_extract_query_ohaeng_list`가 쿼리의 모든 오행을 추출(순서 보존)하고, 열거 필터는 **이름 글자 중 1자 이상의 자원오행이 선택 집합에 포함**되면 통과. 자원오행 기준인 이유: 사용자의 오행 선호는 "부족한 기운 보강" 의도이고, 그 역할은 성명학적으로 자원오행이 담당.

### 4-4. 자연어 입력 UX — B안 채택·구현 (2026-07-07)

자연어 탭을 유지하면서 clarify 실패 모드를 제거했다. 구현하면서 원안("구조화로 완전 변환해 전송")을 한 가지 다듬었다: 자연어를 구조화 요청으로 바꿔 보내면 파서가 못 잡는 자유 조건("ㅅ으로 시작", 순우리말 요청 등)이 소실된다. 그래서 **원문은 그대로 보내되, 실패 원인만 앞뒤에서 제거**하는 방식으로 구현했다 — 목적(자연어 유지 + 실패 제거)은 동일하고 표현력 손실이 없다.

| 구현 | 위치 | 내용 |
|---|---|---|
| 제출 전 인라인 검증 | `NaturalInputForm.tsx` | `parseNameQuery`(미리보기 칩과 동일 파싱)로 **성씨·성별 미포함 시 제출 차단 + 예시 문구 안내** — 서버 clarify의 2대 원인을 입력 단계에서 해소 |
| 대표 성씨 한자 주입 | `fastapi_app/main.py` `_inject_default_surname_hanja` | 자연어 쿼리의 "○씨"에 한자 병기가 없으면 대표 한자 주입("이씨 딸"→"이씨(李) 딸"). **사용자가 이미 한자를 쓴 경우(오행 木火土金水·吉凶 표기 제외)는 건드리지 않음** |
| 예시 칩 정비 | `NaturalInputForm.tsx` | 프롬프트 칩이 검증(성씨+성별)을 통과하는 완성형 예시가 되도록 수정 |

잔여 실패 경로: 대표 한자 맵에 없는 희귀 동음이의 성씨 → 기존 §2-4의 422 안내("한자를 함께 입력해 주세요")가 처리. 순우리말 자연어 요청은 성씨 한자가 불필요하므로 영향 없음(legacy 경로 유지).

### 4-5. Constraint-first 전환 구현 (`naming_graph.py`)

한자 이름 요청의 새 기본 경로 (순우리말·한자+순우리말 동시 요청은 기존 경로 유지):

```
1. _generate_sound_candidates (LLM)  — 트렌드(paper_col) 반영 음절 후보. 유지
2. _enumerate_name_candidates (코드) — (음절 × 한자쌍) 완전 열거·검증:
     · OR 오행 (4-1) · 원·형·이격 大凶 즉시 탈락
     · tier1(4격 모두 길계열) > tier2(흉 포함) > tier3(성씨 획수 미상) 정렬
     · 사용자 오행(4-3) · 획수 범위(4-2) · excludeNames(4-6①)
     · 성씨-첫글자/끝글자 동음 배제 · 성씨 한자 재사용 배제
     · 두음법칙 원음 풀 보강 (용→룡, 이→리, 나→라)
3. _select_candidates (LLM 1회)      — 검증된 후보에서 선택 + 추천 이유 작성.
     LLM 실패 시 점수순 + 템플릿 이유 → 요청 개수 결정론적 보장
4. _render_selected_names (코드)     — 마크다운 답변 + 구조화 결과 동시 렌더.
     모든 수치·오행·뜻이 DB 실측 → 후처리 교정 불필요
5. _verify_names (LLM 1회)           — 부적절 뜻 검출 시 다음 순위 후보로
     결정론적 교체 (LLM 재생성 없음)
6. 후보 부족 시 기존 자유 생성 경로로 자동 fallback
```

효과: 직렬 LLM 호출 최대 7~8회 → **4~5회** (스모크 테스트 실측 14~17초). `/names/generate`는 `structured_results`를 직접 사용해 **2차 변환 LLM 제거** — 변환 환각 경로 소멸.

### 4-6. 백로그 3건 구현

**① excludeNames (이력 중복 회피 복원)**: `/names/generate` body에 `excludeNames: string[]` 추가 (`types.ts` 계약 반영). 열거 단계에서 제외하므로 확실히 배제된다. **남은 프론트 연결**: ResultsScreen "다시 추천"(`onRetry`)에서 현재 결과의 hangul 배열을 담아 재요청 — B 담당자 작업분. queryKey에 요청 객체가 통째로 들어가므로 excludeNames가 다르면 캐시도 자동 분리된다.

**② 뜻 오탈자 교정**: 智 "글기"→"슬기" — ChromaDB hanja_col(문서+메타데이터) + `hanja_documents.json` 2건 교정 완료. 검출: 의심 토큰 전수 스캔 + 상용 인명한자 42자 대조 — 추가 발견 없음. 전수 자동 검증은 국어사전 대조가 필요해 백로그로 유지.

**③ 수리 필터 강도**: 신규 경로는 tier1(4격 모두 길계열) 우선 정렬로 자연 해소 — 실측에서 후보 47건 중 40건이 tier1. 기존 大凶 필터는 fallback 경로 안전망으로 유지.
