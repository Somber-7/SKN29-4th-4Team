`UserConsent` 필드:

- `user`: Django `auth_user`와 1:1 연결

- `privacy_version`
- `terms_agreed_at`
- `privacy_agreed_at`
- `agreed_ip`
- `user_agent`

기존 계정 정보는 별도 사용자 테이블을 새로 만들지 않고 Django 기본 `auth_user`를 사용한다.

| 서비스 의미 | DB 테이블/필드 |
|---|---|
| 이름 | `auth_user.first_name` |
| 아이디 | `auth_user.username` |
| 이메일 | `auth_user.email` |
| 비밀번호 | `auth_user.password` |
| 가입일 | `auth_user.date_joined` |
| 관리자 여부 | `auth_user.is_staff` |

주의:

- 비밀번호는 Django `create_user()`와 `set_password()`를 사용하므로 평문이 아니라 해시로 저장된다.
- 휴대폰 번호는 현재 기획서의 필수 인증 항목이 아니므로 수집/저장하지 않는다.
- 이름과 아이디는 가입자 확인 기준값으로 보고 마이페이지에서 수정하지 않는다.
- 이메일은 안내 수신용 정보로 보고 마이페이지에서 수정 가능하게 했다.

## 8. 팀원 공유사항

### 8.1 다른 담당자와 겹치지 않도록 지켜야 할 선

- B 담당자는 `ResultsScreen`에서 작명 결과 저장 시점에 `POST /api/me/history`를 연결하면 된다.
- A 작업은 이미 `/api/me/history` 조회와 백엔드 API 계약을 유지하고 있으므로, B 작업자는 저장 호출만 추가하면 된다.
- D 담당자는 React 관리자 화면과 `/api/admin/*` API를 별도 범위에서 진행하면 된다.
- 이번 작업은 관리자용 사용자 CRUD가 아니라 사용자 본인의 계정 CRUD다.
- Insights/FAQ 실 API 전환은 이번 작업에서 제외했다.

### 8.2 충돌 가능 접점

- `frontend/src/app/components/layout/GNB.tsx`
  - 마이페이지 드롭다운을 추가했다.
  - 관리자/공용 라우팅 담당자가 GNB를 수정할 경우 충돌 가능성이 있다.
- `frontend/src/app/types.ts`
  - `AuthUser.username`, `MyPageSection`이 추가되었다.
- `frontend/src/api/client.ts`
  - CSRF 토큰은 고정 저장하지 않고 매 요청마다 쿠키에서 읽는다.
  - 로그인 후 CSRF 토큰이 회전될 수 있으므로 이 방식이 필요하다.
- `webapp/naming/views.py`
  - `/api/me`는 `GET`, `PATCH`, `DELETE`를 함께 처리한다.
  - `history_view`의 `POST` 계약은 유지했지만 프론트 저장 호출은 B 담당이다.

4. 관리자 화면에서 사용자를 삭제/수정하는 기능을 만들 경우 팀장과 정책을 먼저 맞춘다.
   - 일반 사용자 본인 탈퇴: A 작업 범위
   - 관리자의 사용자 관리: D 작업 범위
5. GNB나 라우터를 수정할 때 `마이페이지` 드롭다운과 `/mypage?section=...` 이동이 깨지지 않는지 확인한다.
6. 관리자 화면의 route 변경은 `router.tsx`, `useCurrentScreen`, `useScreenNav`를 함께 확인한다.

#### 팀장 / 통합 담당자가 해야 할 일

1. PR 머지 전 `data/chroma/chroma.sqlite3`가 커밋에 들어가지 않았는지 확인한다.
2. `.env.local`이 커밋에 들어가지 않았는지 확인한다.
3. AWS 배포 전 Django 마이그레이션을 적용한다.
   ```bash
   python manage.py migrate
   ```
4. 배포 빌드에서 `VITE_USE_MOCK_AUTH=true`가 켜져 있지 않은지 확인한다.
5. 배포 후 브라우저에서 아래를 직접 확인한다.
   - 회원가입 약관/개인정보처리방침 모달 문구
   - 아이디 로그인
   - 비밀번호 재설정
   - 마이페이지 이메일 수정
   - 회원 탈퇴
6. 배포 서버에서 nginx 또는 정적 파일 캐시 때문에 이전 프론트 번들이 보이지 않는지 확인한다.

#### 모든 팀원이 공통으로 지켜야 할 일

1. `git add .`로 한 번에 올리지 않는다.
2. 본인 담당 범위 밖 파일을 수정해야 하면 작업내역 MD에 이유를 남긴다.
3. 빌드/타입체크가 통과해도 경고가 있으면 원인을 확인한다.
4. Docker 실행으로 바뀐 로컬 DB/Chroma 파일은 커밋하지 않는다.
5. API 계약을 바꾸면 프론트/백엔드/문서를 같이 맞춘다.
