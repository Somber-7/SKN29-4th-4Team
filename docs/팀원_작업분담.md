# 4차 프로젝트 — 다음 단계 작업분담 (A · B · D)

> 프론트-백엔드 기본 통합(STEP 2~6: nginx 라우팅, FastAPI `/names/generate`, Django 인증 API, `NamingHistory` 모델)은 이미 `main`에 병합·배포되어 있습니다. 이 문서는 그 다음 단계로 남은 작업을 3명이 나눠서 진행하기 위한 범위 정리입니다.
>
> 작업 방식은 [팀원_작업_매뉴얼.md](./팀원_작업_매뉴얼.md)와 동일합니다: 각자 브랜치에서 작업 → 로컬(`docker-compose.local.yml`)에서 테스트 → `origin/<본인브랜치>`에 push → Pull Request → 팀장이 리뷰 후 `main` 병합 → 자동 배포.

---

## A. 인증 연동 (로그인 · 회원가입 · 마이페이지)

**현재 상태:** 백엔드 API(`/api/auth/login`, `/signup`, `/me`, `/me/change-password` 등)는 이미 동작하고, 프론트의 `authApi`(`frontend/src/api/auth.ts`)에도 실 API를 호출하는 `realAdapter`가 이미 구현되어 있습니다. 그런데 화면들이 이걸 아직 호출하지 않고 있습니다.

| 문제 | 위치 |
|---|---|
| `AuthProvider`가 서버 세션이 아니라 `sessionStorage` 데모 패턴으로만 동작 (`TODO(API)` 주석 있음) | `frontend/src/app/providers/AuthProvider.tsx` |
| `me_view`가 `PATCH`/`DELETE`만 지원 — 새로고침 시 "지금 로그인 상태인지" 확인할 GET 방법이 없음 | `webapp/naming/views.py` (`@require_http_methods(["PATCH", "DELETE"])`) |
| `LoginScreen`이 `authApi.login()`을 호출하지 않고 하드코딩된 데모 계정 체크만 함 | `frontend/src/app/screens/LoginScreen.tsx` (76번째 줄 `TODO(API)` 주석) |
| `SignupScreen`/`MyPageScreen`도 마찬가지로 `authApi.signup/updateProfile/changePassword/withdraw` 미연결 | 각 스크린 파일 |
| 앱 로드 시 CSRF 쿠키(`GET /api/auth/csrf`)를 받아오는 호출이 없음 | 신규 추가 필요 |

**할 일**
1. `webapp/naming/views.py`의 `me_view`에 `GET` 메서드 추가 — 현재 세션의 사용자 정보 반환 (데코레이터를 `["GET", "PATCH", "DELETE"]`로).
2. `AuthProvider.tsx`를 앱 마운트 시 위 GET 엔드포인트로 세션을 확인하는 방식으로 재작성 (React Query `useQuery` 권장, 기존 `frontend/src/app/hooks/` 패턴 참고).
3. `LoginScreen`/`SignupScreen`/`MyPageScreen`의 `TODO(API)` 부분을 `authApi.login/signup/updateProfile/changePassword/withdraw` 실호출로 교체.
4. 앱 시작 시점(예: `App.tsx`나 `AuthProvider` 마운트 시)에 `GET /api/auth/csrf` 한 번 호출.

## B. 작명 생성 → 히스토리 저장 연동

**현재 상태:** `useNameResults` 훅(`frontend/src/app/hooks/useNameResults.ts`)은 이미 실제 `/naming-api/names/generate`를 호출합니다. 문제는 그 결과를 히스토리에 저장하는 연동이 통째로 빠져 있다는 것입니다 — 백엔드 `POST /api/me/history`는 이미 동작하는데 프론트가 호출하지 않습니다.

| 문제 | 위치 |
|---|---|
| 결과 생성 후 히스토리 저장 호출이 없음 | `frontend/src/app/screens/ResultsScreen.tsx` |
| `ProcessingScreen`의 가짜 진행바가 mock의 짧은 지연 기준으로 만들어져 있어, 실제 LLM 응답(수십 초, 최대 90초)과 타이밍이 안 맞음 | `frontend/src/app/screens/ProcessingScreen.tsx` |

**할 일**
1. `ResultsScreen`에서 작명 결과가 확정되는 시점에 `authApi`(또는 새 `historyApi`)를 통해 `POST /api/me/history`를 호출해서 저장 (`query`, `request`, `results` 바디 — `webapp/naming/views.py`의 `history_view` POST 분기 참고).
2. `ProcessingScreen`의 진행바/문구 타이밍을 실제 응답 시간(최대 90초, FastAPI 쪽 timeout 기준)에 맞게 조정.
3. `HistoryScreen`/`MyPageScreen`에서 실제로 저장된 기록이 잘 보이는지 검증 (`useHistory` 훅은 이미 실 API 연결되어 있음).

## D. 관리자 화면 실 연동

**현재 상태:** React 관리자 화면 4개(Dashboard/Content/Users/Settings)가 전부 mock 데이터로만 동작합니다. `adminApi`(`frontend/src/api/admin.ts`)에 Dashboard/Content/Users용 API 계약은 이미 있지만(`GET /api/admin/dashboard|content|users`), Django 쪽에 대응하는 뷰가 없습니다. Settings는 계약 자체가 없어 새로 설계해야 합니다.

**라우트 변경 (먼저 처리):** 현재 경로 `/adminDashboard`, `/adminContent`, `/adminUsers`, `/adminSettings`를 `/manage/dashboard`, `/manage/content`, `/manage/users`, `/manage/settings`로 변경합니다 (`/admin`으로 시작하는 경로는 nginx의 `location /admin/`—Django Admin용—과 헷갈리기 쉬워서 분리). 수정할 곳 3군데:
- `frontend/src/app/router.tsx` (303~334번째 줄 근처, `path=` 4곳)
- `frontend/src/app/hooks/useCurrentScreen.ts` (`PATH_TO_SCREEN` 매핑)
- `frontend/src/app/hooks/useScreenNav.ts` (`SCREEN_PATHS` 매핑)

**백엔드 신규 작업**
1. `SiteSettings` 모델 신설 (`webapp/naming/models.py`) — 서비스명, 지원 이메일, 기본 결과 개수, 한자/수리/법령/논문 출처 표시 토글 4개, 점검 모드. 단일 row로 운용(사이트 전체 설정 1건).
2. `GET/PUT /api/admin/settings` 뷰 추가.
3. `GET /api/admin/dashboard` — `NamingHistory` 기반 통계(총 요청 수 등), 주간 요청 추이, 출처 분포(결과의 `sources` 필드 집계), 최근 요청 목록.
4. `GET /api/admin/content` — 인명용 한자 목록 + 출처 목록. **주의:** 이 데이터는 Django ORM에 없고 ChromaDB(`data/chroma`)·Neo4j에 있음 — 조회 방식을 새로 설계해야 함 (FastAPI를 거쳐 가져오거나, Django에서 직접 Neo4j에 연결하거나 결정 필요). D 작업 중 가장 손이 많이 가는 부분.
5. `GET /api/admin/users` — Django `User` + 사용자별 `NamingHistory` 개수 조합.

**프론트 신규 작업**
1. `frontend/src/api/admin.ts`에 `getSettings`/`updateSettings` 계약 추가.
2. `AdminSettingsScreen`을 로컬 `useState` 전용에서 `adminApi` 연동으로 교체.
3. `AdminDashboardScreen`/`AdminContentScreen`/`AdminUsersScreen`은 이미 `adminApi` 훅을 쓰고 있으므로, 백엔드 완성 후 `VITE_USE_MOCK=false`로 확인.

## 참고: Insights · FAQ는 계속 mock

관리자 화면이 D로 실 연동되면서, 범위에서 제외된 화면은 Insights(이름 트렌드)·FAQ 2개만 남습니다. 이 둘은 이번 단계에서 백엔드 작업 대상이 아니므로 `VITE_USE_MOCK=false`로 전환해도 계속 mock을 쓰도록 개별 처리가 필요합니다 (전역 플래그 하나라 그대로 두면 이 화면들의 API 호출이 404로 깨짐) — 이 부분은 통합 검증 담당(팀장)이 처리합니다.
